# 需求取捨決策

> 這份文件可以由 Codex 先產生草稿，但最後必須由你刪除、修改或補充。請不要讓 AI 替你做最後決策。

## 我決定 v1 優先服務的使用者

- [x] 資訊整理者

原因：

我的系統最終服務的是災民，但 v1 不直接解決完整救助流程。v1 先處理災民資訊被中間人整理時最容易出錯的一段：原始求助如何被保留、標示不確定性、避免被誤轉成可派工任務。

所以 v1 的直接使用者是資訊整理者，最終受益者是災民。這不是改變服務對象，而是先修正災民需求被轉交過程中的失真風險。

這個決定代表：v1 的重點不是「做出任務」，而是「讓災民的需求不要在整理過程中被改錯、補錯、判斷錯」。三個 persona 都指出目前「任務欄」「高可信度」「可直接使用」等字眼會把草稿推向可行動語境，因此 v1 應先讓資訊整理者清楚分辨「原始資訊」「AI 判斷草稿」「人類確認狀態」與「不能直接變成任務的理由」。

這是一個刻意的分階段取捨：先把中間層做對，才有資格談下一步怎麼讓災民真正更快、更安全地得到協助。否則系統看起來像在服務災民，實際上可能只是更快地把災民的不完整求助轉成錯誤任務。

## 我決定這個系統的上層目標

- [x] 讓災民需求在被轉交、整理、判斷時，不失真、不被過度推測、不被太早當成可派工任務。
- [x] 讓後續協作者知道哪些內容是災民或回報者原話，哪些是 AI 草稿，哪些是人類判斷。
- [x] 讓資訊不足的求助不會被假裝完整，也不會因為不好整理就消失。

我現在不把成功定義成「產生越多任務越好」。比較合理的成功定義是：

> 災民的需求被保留原意、被安全轉交、被清楚標示不確定性，並且沒有被 AI 或中間人誤改成錯誤行動。

這也表示 v1 的核心不是派工效率，而是資訊轉交的安全性。

## 我需要看見的資訊生命週期

一筆資訊不應該直接從「有人回報」跳到「志工任務」。比較安全的生命週期應該是：

```text
災民處境
→ 原始回報
→ 待確認線索
→ 人工整理
→ 候選行動
→ 可派工任務
```

目前 v1 應該先處理前四段，尤其是「原始回報 → 待確認線索 → 人工整理」。至於「候選行動 → 可派工任務」，需要更明確的人類確認規則，不能由 AI 或簡單表單自動推進。

這裡還需要補的決策：

- 誰可以把「待確認線索」推進成「候選行動」？
- 什麼資訊不足時，必須回頭問災民或代填者？
- 哪些情況下，資訊只能暫停，不能變成任務？
- 災民是否會知道自己的需求目前停在哪個階段？
- 如果資訊整理者判斷「不能派工」，災民的需求會不會因此消失？

## 我決定災民填表體驗要先簡單

- [x] 表單選項要簡潔、清楚，讓災民可以快速選到常見災害或常見困難。
- [x] 如果選項沒有符合情境，災民可以選「其他」，再用文字描述。
- [x] 表單不應該要求災民一開始就理解完整救災流程、任務分類或志工分派邏輯。
- [x] 生命安全相關情境需要補充備註，讓資訊整理者能判斷嚴重程度與下一步確認方式。

原因：

如果我是災民，我不會希望表單太複雜。當下最重要的是快速留下求助資訊，而不是把自己的處境翻譯成系統內部分類。表單應該先提供常見選項，讓災民用最少負擔完成回報；只有選項不符合時，才需要用「其他」補充文字。

表單可以先問幾個簡單但有用的問題：

- 你現在遇到了什麼困難？
- 你需要哪一類人或哪一種專業協助？
- 這件事是否和生命安全有關？
- 你大概在哪裡？如果不知道完整地點，請描述你迷失、受困或附近看得到的線索。
- 你受困或需要協助大約多久了？
- 有沒有必須補充的情境細節？

如果情境和生命安全有關，就不能只靠選項。災民需要用備註補充更細的情境，例如受困時間、大概位置、身體狀況、周遭環境或是否有人同行。這些內容不是為了讓 AI 自動判斷優先順序，而是讓資訊整理者有足夠脈絡做人工確認。

## 我決定送出後要有審核回覆

- [x] 災民送出後，系統應該讓送出者知道資料已送出並進入審核。
- [x] 審核完成後，系統應該回覆送出者審核結果，例如「審核成功」或「需要補充資訊」。
- [x] 「審核成功」不等於已經派工，也不等於一定有人會立刻到場。

原因：

災民送出需求後，不能只看到表單消失。至少要知道系統有沒有收到、目前是不是在審核、是否需要補資料。這會降低求助者的不安，也避免重複送出或以為已經有人出發。

但這裡要清楚區分：

- 已送出：系統收到原始回報。
- 審核中：資訊整理者正在確認內容是否足夠。
- 審核成功：資料已可進入下一步處理。
- 需要補充：目前資訊不足，不能安全往下轉成候選行動。

「審核成功」只能代表資料通過整理或確認門檻，不能直接顯示成「已派工」。

## 我決定回報表單是誰在填

- [x] 災民本人（或代災民填寫的人），不是志工。
- [x] 志工只在「志工任務欄」看資訊、決定要不要回應，志工本身不填這張表單。

原因：

`interview-notes.md` 裡資訊整理者和行動者都反問過「這個表單是給回報者填、還是給志工整理資料用」，一直沒有定論。現在確定填表的人是災民本人或代災民填寫的人，這代表表單不是志工報名表，也不是資訊整理者的內部工作表。它的任務是讓災民處境被安全留下來。

所以表單欄位不能先問「志工怎麼做」，而要先問「災民現在遇到什麼、需要什麼、什麼資訊還不確定」。

這代表現有欄位裡有幾個是寫反的，需要重新設計，不能照抄志工報名表的語氣：

- 「可以來幫忙的時間」「要來幫忙什麼東西」（`TrustAssessmentPage.tsx` 表單欄位）目前問的是「你何時有空來幫忙、你會帶什麼來」，這是志工的語氣，不是災民描述自己需求的語氣，必須改寫成「你需要什麼時候得到協助」「你需要什麼物資」。
- 「志工任務類型」「建議志工特長」這兩欄，如果是災民填，意義應該是「災民覺得自己需要哪種協助」，不是志工自己選任務類型；文案需要調整成災民能理解、能回答的問法。
- 「信心程度」原本設計是回報者對自己說法的把握程度；如果填表的是災民本人描述自己的處境，這一欄的意義要重新想。災民對自己的需求通常不需要自評信心，這欄可能該拿掉，或改成「這個狀況是現在進行式，還是已經緩解」這類跟時效有關的問題。

這幾個欄位在確認「填表人是誰」之前沒辦法改對，所以這個決定要先拍板，才能動手調整表單。

但這仍然只是表單層級的修正。更上層的判斷是：表單應該保護災民原意，不應該逼災民把處境翻譯成志工任務。

## 這個使用者最想完成什麼

- 快速判斷哪些資訊值得進一步確認，哪些不能直接變成任務。
- 保留原始資訊與判斷理由，讓下一位協作者知道為什麼這筆資料沒有被直接採用。
- 看出哪些內容是原文、哪些是 AI 依內容特徵產生的草稿、哪些仍需要人類逐筆覆核。
- 看出災民需求目前卡在哪一段：缺資料、待確認、已整理、還是可以進入候選行動。
- 避免把災民的模糊求助整理得太漂亮，讓下一位協作者誤以為資訊已經完整。
- 用簡單選項快速理解災民遇到的困難，並在生命安全相關情境看到必要備註。
- 在審核後給送出者清楚回覆，避免災民不知道自己的求助是否被接住。

## 這個使用者最怕什麼錯誤

- 把未確認資訊當成已確認事實。
- 讓不夠完整的資訊被誤認為可直接行動的任務。
- 把表單輸入的「信心程度」或規則分數誤讀成查核後的可信度。
- 讓手動新增的任務內容脫離原始資訊與來源，下一位協作者看不出它是否被確認。
- 因為資料不完整，就讓災民需求被忽略或消失。
- 為了讓資料看起來能派工，而把災民原本沒說的內容補進去。
- 表單過度複雜，讓災民在緊急或壓力大的狀態下放棄填寫。
- 系統顯示「審核成功」時，被誤解成已經有人出發或已經完成派工。

## 我今天暫時不處理的需求

- 我暫時不處理自動補齊資料欄位、地點比對或任務自動派工的功能。
- 我也不會在 v1 中把所有資訊都做成完整工作流，因為這會讓使用者把介面誤以為系統已經替他做了判斷。
- 我暫時不把 v1 做成完整的災民求助系統，也不把它做成行動者直接接任務的畫面。
- 我可以先定義災民送出與審核回覆的基本狀態，但暫時不處理完整追蹤、補充資料、通知與派工流程。
- 我暫時不決定志工如何認領任務，因為目前還沒有足夠規則判斷什麼資訊可以變成任務。

## 我不採用的 AI 建議

- AI 建議：把不確定資訊自動轉成任務草稿。
  - 不採用原因：這會混淆「線索」與「可行動任務」，並把缺乏確認的資訊當成已確認決策。
- AI 建議：用表單的信心程度或簡單規則顯示「高可信度 / 中可信度」。
  - 不採用原因：使用者可能把它誤認為查核結果；v1 應改用更保守的語言，例如「回報者自評信心」或「規則草稿分數」，並與查核狀態分開。
- AI 建議：只要把欄位文案改成災民語氣，就等於開始服務災民。
  - 不採用原因：文案只是入口表層。真正服務災民，還要決定資訊如何被接住、誰回頭確認、資訊不足時怎麼辦、災民是否知道處理狀態。
- AI 建議：用災害類型選項直接推導協助等級。
  - 不採用原因：常見選項只能降低填寫負擔，不能取代人工判斷。生命安全、受困時間、地點不明等情境仍需要備註與人工審核。

## 必須保留人工確認的判斷

- 任何來源不明、轉述、缺少時間或地點、或標記為待確認的資訊，都必須保留人工確認流程。
- 任何可能與其他資料衝突、過期或缺乏原始證據的資訊，都不能被直接視為可派工。
- 任何由 AI 依內容特徵產生的判斷理由，都只能當成整理草稿，不能當成正式結論。
- 任何被放進任務語境的內容，都必須能回到原始資訊與確認狀態；如果做不到，就不應顯示成任務。
- 任何會改變災民原意的整理，都必須留下原文對照與人工確認痕跡。
- 任何把「需求」轉成「行動」的決定，都必須由人類確認，不能由 AI 或欄位分數自動決定。
- 任何生命安全相關情境，都不能只看選項判斷，必須閱讀備註與原始描述。
- 任何審核結果都要避免讓災民誤以為已經派工，除非真的有後續派工確認。

## 今天明確不做什麼

- 不做後端或資料庫。
- 不做真實地圖、外部 API 或真實救災判斷。
- 不把未確認資訊包裝成已確認內容。
- 不把 v1 做成正式派工系統。
- 不讓 AI 自動判斷「可以派工」。
- 不替災民決定他真正需要什麼。
- 不把災民的模糊求助自動翻譯成完整任務。
- 不因為資訊不足就讓災民需求消失。
- 不讓志工看到看似完整但其實未確認的任務。
- 不把「審核成功」寫成「救援已安排」。
- 不要求災民理解志工任務分類才能送出求助。

## 我仍不確定的地方

- 我還不確定要在每筆資料上展示多少判斷理由，才能讓資訊整理者安心，但又不讓畫面過於冗長。
- 我還不確定 v1 應該完全移除任務欄，或保留但改名為「候選線索草稿」並加上更明確的未確認標示。
- 我還不確定下一階段要怎麼補上災民視角：是直接訪談一個「災民」persona，還是先確認資訊整理者的判斷流程，真的會讓災民端得到更快、更正確的回應，再決定要不要為 v1 加入災民視角。
- 我還不確定災民送出需求後，是否需要看到處理進度。如果需要，v1 只做資訊整理就不夠，後面必須設計回饋機制。
- 我還不確定資訊整理者判斷「不能派工」時，災民需求應該被放在哪裡。它不應該變成任務，但也不應該消失。
- 我還不確定「代災民填寫的人」要如何標示，因為代填會增加轉述風險，但也可能是災民最容易留下需求的方式。
- 我還不確定常見災害或常見困難的選項應該有哪些，這需要再用災民視角檢查，避免選項看似完整但實際不符合求助情境。
- 我還不確定哪些情境必須強制填備註，例如生命安全、地點不明、受困時間過長、需要特定專業人員等。
