# Main Agent 訪談彙整

> 這份文件可以由 Codex 先產生草稿，但你必須刪除、修改或補充。請保留人類確認痕跡。

## 三個 persona 的共同需求

- 使用者明確說的：三個角色都想分清楚「線索」「已確認資訊」「可行動任務」之間的差異。
- 使用者明確說的：三個角色都擔心「任務欄」「新增到任務欄」「高可信度」這些詞會讓人誤以為資料已經可以派工。
- 我推論的：v1 需要把資訊品質和任務行動性拆開呈現，不能只靠一個可信度標籤解決。

## 三個 persona 之間的衝突需求

- 使用者明確說的：回報者想快速留下不完整資訊，整理者想保留原文與判斷脈絡，行動者想快速知道能不能出發。
- 使用者明確說的：回報者不想被迫硬填不確定欄位，但整理者和行動者需要看到不確定處，才不會誤用。
- 我推論的：v1 若同時服務三種角色，會把畫面拉向不同方向；需要先選一個主要使用者。

## Phase 0 原型中被推翻的假設

- 原本假設：只要標示來源與查核狀態，使用者就能理解資料能不能使用。
  - 被什麼回饋推翻：三個 persona 都提到「可直接使用：是」和「可直接變成任務：否」會讓人搞不清楚能不能行動。
- 原本假設：「任務欄」只是 Phase 0 練習用草稿區，使用者會自然理解不是正式派工。
  - 被什麼回饋推翻：三個 persona 都把「任務欄」讀成可能可以派人的區域，尤其行動者明確擔心自己會直接出發。
- 原本假設：信心程度可以用來幫助排序或描述草稿。
  - 被什麼回饋推翻：回報者與整理者都指出，表單的「信心程度」若被顯示成「高可信度」，會被誤讀成查核結果。

## 最容易造成誤導的風險

- 把來源類型誤讀成查核結果，讓使用者以為「看起來像官方」就等於「可以採用」。
- 把表單輸入或規則分數誤讀成「查核後可信度」，特別是任務卡上的「高可信度 / 中可信度 / 低可信度」。
- 讓未確認資訊過早進入任務或派工語境，造成行動者誤以為可以直接採取下一步。
- 讓任務欄裡的手動新增內容脫離原始資訊，看不到來源與人工覆核狀態。

## 可以考慮的設計方向

> 這裡只是可考慮方向，不是標準答案。

- 把「原始資訊」「AI 判斷草稿」「人類覆核狀態」分開標示，避免下一位協作者分不清哪個是原文、哪個是推論。
- 把「可作為線索」和「可採取行動」明確拆開，不再使用容易混淆的「可直接使用」。
- 若保留任務相關區塊，應把它命名為「整理草稿」或「候選線索」，並明確標示不是已確認任務。
- 在與任務、志工分派、可信度相關的文案旁，顯示來源與人工確認狀態。

## 需要人類決策的取捨

- 取捨 1：
  - 選項 A：把畫面做得更像回報入口，讓回報者快速送出不完整資訊。
  - 選項 B：把畫面做成資訊整理者的工作台，保留原文、AI 草稿、人工確認狀態與不能變成任務的理由。
  - 我目前傾向：選項 B。
  - 理由：這個階段最重要的不是讓所有人都更快填表，而是避免把不確定資訊誤當成可採取行動的資訊。
- 取捨 2：
  - 選項 A：保留任務欄，讓使用者可以練習把資訊整理成行動草稿。
  - 選項 B：先移除或降級任務欄語境，改成「候選線索」或「需要確認事項」。
  - 我目前傾向：選項 B。
  - 理由：三個 persona 都指出「任務欄」容易造成誤讀，這是目前最明確的風險。

## 不應該讓 AI 自動決定的事情

- 不應該讓 AI 自動把不確定報告改寫成已確認任務。
- 不應該讓 AI 自動補齊缺失的地點、時間或人員資訊，尤其在資料來源本身不夠可靠時。
- 不應該讓 AI 把「可能」或「轉述」的語氣直接改成「已確認」。
- 不應該讓 AI 自動把表單中的信心程度轉成查核可信度。
- 不應該讓 AI 自動判定某筆候選線索可以派工。

## 我採用的 AI 整理

- 我採用 sub-agent 對三個 persona 的模擬回饋，特別是「任務欄會讓人誤以為可以派工」與「信心程度被顯示成可信度」這兩個風險。
- 我採用 main agent 對共通需求、衝突需求與風險的整理，但仍把它標成需求分析草稿。

## 我不採用的 AI 建議

- 建議：把所有不確定資訊自動整理成可派工任務草稿，或用信心程度自動推導可信度。
  - 不採用原因：這會把尚未確認的內容誤導成可行動決策，與本階段的安全原則相衝。

## 我自己補充的判斷

- v1 的核心不是做出完整工作流，而是讓使用者更清楚地看見「資訊品質」與「行動性」之間的差異。
- 這個階段應該優先服務資訊整理者，因為他們最需要判斷哪些資訊可以往下推，哪些應該被暫停。
- 我認為目前最大問題不是缺少功能，而是某些文案把草稿推向「任務」語境，讓使用者容易過度信任。
- 我原本說「系統主要服務災民」，但重新檢視這次訪談才發現：回報者、資訊整理者、行動者三個 persona 沒有一個是災民本人，選資訊整理者當 v1 優先使用者，等於是選了一個離災民最遠的中間角色。這不代表決定錯了——先讓中間層不誤判，確實是讓下游更安全服務災民的前提——但這是一個需要講清楚的取捨，不能讓「服務災民」這句話蓋過「v1 實際上服務的是誰」。

## 我仍不確定的地方

- 我們應該在每筆資料上展示多少證據與理由，才足夠讓人安心但又不讓畫面太重。
- 我還不確定 v1 是否應完全移除任務欄，或保留但改成非常明確的「不可派工草稿」。
- 我還不確定：只服務資訊整理者，會不會讓災民的需求被排到更後面才被人看見？我需要找方法確認，資訊整理者做出的判斷最終有沒有真的讓災民更快、更安全地得到協助，而不是只讓後台流程看起來更嚴謹。

## 小隊討論後的補充

- 我們應該把「需要人工確認」與「不能直接變成任務」這兩個概念在畫面上更明確地分開。
- 待小隊補充：哪個 sub-agent 的回饋最具體？哪個回饋太像產品顧問？main agent 有沒有過度簡化？
