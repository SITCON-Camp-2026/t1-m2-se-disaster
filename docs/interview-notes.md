# 使用者訪談紀錄

> 這份文件可以由 Codex 先產生草稿，但你必須刪除、修改或補充。請不要直接把 AI 輸出當成最後結論。

## 我的 Phase 0 prototype 現況

目前的原型把原始資訊、可信度評估與整理工作台分成三個區塊，讓使用者能先看來源、再看判斷、最後試著整理成可行動的草稿。畫面已經把來源類型與查核狀態呈現出來，也提供了一個簡化的志工任務欄，方便大家討論哪些資訊可以進一步處理。可是這個版本仍然把「判斷」和「可操作任務」放得太近，對第一次接觸的人來說，容易把草稿當成已確認內容。

## 訪談方式

- 使用 `release-packs/01-interview-kit/docs/releases/01-interview-kit.md` 與三份 persona 檔案。
- 啟用三個 sub-agent 分別模擬回報者、資訊整理者、行動者。
- sub-agent 只扮演使用者給回饋，不改程式，也不替小組做產品決策。
- 這些回饋仍是 AI 模擬訪談草稿，需要人類刪改與確認。

## Persona 1：回報者

### 我想完成的事

- 我想很快把我知道或聽到的狀況送出去，就算資訊還不完整。
- 如果我是轉述別人的話，我希望系統不要把我當成親眼確認的人。

### 我看不懂的地方

- 「可信度判斷」裡同時有「志工任務欄」和「難民現場回報」，我不確定這是給回報者填，還是給志工整理資料用。
- 「可直接使用：是」但又說「可直接變成任務：否」，我會搞不清楚到底能不能拿去行動。

### 我最怕的錯誤

- 我只是先回報一個線索，但按下「新增到任務欄」後，看起來像已經變成可以派人的任務。
- 我填「信心程度：中」或「高」，系統就把它顯示成「高可信度」，讓別人以為這是真的被查核過。

### 我不敢直接相信的資訊

- 「官方公告」「現場回報」這些來源名稱，我不敢只看名稱就相信，因為我不知道它是不是最新、是不是完整。
- 原始資訊裡如果只有模糊地點、轉述、或「聽說」，我不敢相信它可以直接拿來派工。

### 我希望系統提醒我的事

- 在我新增前，提醒我：這只是回報或線索，不代表已確認，也不一定會直接變成任務。
- 如果我填了不確定的地點、時間、數量或需求，提醒我可以保留不確定，不要硬填。

### 我想反問開發者的一個問題

- 我送出回報後，下一個看到的人會清楚知道「我是親眼看到」還是「我只是轉述」嗎？

## Persona 2：資訊整理者

### 我想完成的事

- 我想先分清楚「原始資訊」「AI 判斷草稿」「人類已確認判斷」各是哪一種。
- 我想把不能直接變成任務的原因留下來，讓下一位協作者不要誤用。

### 我看不懂的地方

- 「可信度判斷」頁面一邊說需要人類覆核，一邊又有「志工任務欄」和「新增到任務欄」，我不確定這是不是已經可以派工。
- 「可直接使用：是」但「可直接變成任務：否」對我來說有點危險，我會想知道這裡的「使用」到底是只能當線索，還是可以拿去安排人。

### 我最怕的錯誤

- 我最怕有人看到「高可信度」「中可信度」就以為這是查核結果，而不是表單輸入或規則分數。
- 我也怕「官方公告」「已確認」這類字樣讓人忽略時效、原文脈絡，直接把它當成現在仍有效。

### 我不敢直接相信的資訊

- 我不敢直接相信由 agent 依內容特徵產生的判斷理由，因為我看不到人類是否已經逐筆覆核。
- 我不敢直接相信任務欄裡手動新增的內容，因為它看起來可能脫離原始資訊，也沒有明確連回來源。

### 我希望系統提醒我的事

- 當我把任何資料送進整理工作台或任務欄時，希望畫面提醒我：這不代表已確認，也不代表可派工。
- 當畫面出現「任務」「志工分派」「高可信度」這些詞時，希望提醒我這些可能只是整理中的草稿。

### 我想反問開發者的一個問題

- 你希望我在這個 prototype 裡扮演的是「整理原始資訊的人」，還是「建立志工任務的人」？現在畫面讓我覺得兩個角色混在一起。

## Persona 3：行動者

### 我想完成的事

- 我想先分清楚：哪些只是線索，哪些真的可以讓我下一步行動。
- 我想避免看到「任務欄」就以為我可以直接出發。

### 我看不懂的地方

- 「可作為線索」旁邊又寫「可直接使用：是」，我會不確定這是不是代表可以行動。
- 「高可信度 / 中可信度 / 低可信度」出現在任務卡上，但我看不出它是誰判斷的，跟查核狀態是不是同一件事。

### 我最怕的錯誤

- 我怕把表單新增出來的任務，誤以為已經被現場確認。
- 我怕「官方公告」或「現場回報」這種來源名稱，讓我下意識覺得比較可靠，但其實內容還沒確認。

### 我不敢直接相信的資訊

- 我不敢直接相信任何只有社群、電話、轉述來源的內容。
- 我也不敢直接相信地點、時間、需求物資，因為畫面有提醒可能過期或不完整。

### 我希望系統提醒我的事

- 當我準備把資訊變成任務時，希望它明確提醒：這還不是已確認任務。
- 如果資料可能過期、互相衝突、或地點不夠清楚，希望不要只藏在說明文字裡。

### 我想反問開發者的一個問題

- 這個畫面裡的「任務欄」是給我真的採取行動用，還是只是練習整理資訊的草稿區？

## 人類補充

請寫下 Codex 沒有注意到，但你覺得重要的觀察。

- 目前的畫面很容易讓人把「判斷摘要」誤讀成「已經有結論」，所以 v1 需要更明確地把「待確認」和「可採取行動」分開。
- 如果我們要讓使用者信任這個工具，必須把原始文字、判斷理由與下一步建議放在同一個視角裡，而不是把它們分散在不同區塊。
- 我發現這三個 persona（回報者、資訊整理者、行動者）都不是災民本人——回報者是轉述或代填的人，資訊整理者是後台判斷的人，行動者是決定要不要出發的志工。如果我的系統最終要服務的是災民，這次訪談其實還沒有問過「災民自己最想完成什麼、最怕系統對他做錯什麼」。這是這份訪談紀錄目前最大的缺口，不是可以忽略的小事，需要在後續補一輪災民視角的訪談，或至少在 `decisions.md` 裡講清楚 v1 為什麼暫時不直接訪談災民。
